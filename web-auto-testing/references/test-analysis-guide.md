# 测试分析指南

本文档提供系统化的测试分析方法论，帮助设计全面、有效的测试用例。

## 1. 测试类型选择指南

### 1.1 测试类型定义

| 测试类型 | 定义 | 适用场景 | 覆盖范围 | 用例规模 | 执行时间 |
|---------|------|---------|---------|---------|---------|
| **Smoke（冒烟测试）** | 验证核心功能是否可用的快速测试 | 新版本部署后、快速验证 | 核心业务流程、关键路径 | 核心流程（最少化） | 5-15 分钟 |
| **单功能测试** | 完整测试指定功能的所有场景 | 用户说"测试XX功能"时 | 指定功能的正向+异常+边界 | 每个功能点×场景类型 | 15-30 分钟 |
| **回归测试** | 验证现有功能未受新代码影响的测试 | 代码变更后、发版前 | 变更功能+高频功能+历史问题 | 变更功能点+依赖功能 | 20-40 分钟 |
| **完整测试** | 全面验证所有功能的测试（含跨系统场景） | 发版前、重大变更后 | 所有功能 + 跨系统场景 + 边界场景 | 所有功能点×场景类型 + 跨系统流程 | 1-3 小时 |
| **E2E（端到端测试）** | 验证跨系统、跨模块完整流程的测试 | 集成测试、用户验收测试 | 跨系统业务流程 | 业务流程数×关键节点 | 30-60 分钟 |

**用例规模说明**：
- **核心流程（最少化）**：仅主流程，最少用例
- **每个功能点×场景类型**：按识别的功能点数乘以场景类型（正向、异常、边界）
- **变更功能点+依赖功能**：本次变更的功能加上可能受影响的依赖功能
- **所有功能点×场景类型**：应用内所有功能点，每种功能点覆盖正向、异常、边界场景
- **业务流程数×关键节点**：每个业务流程的关键验证点

### 1.2 选择决策树

```
开始
  │
  ├─ 是否验证跨系统流程？
  │   └─ 是 → E2E 测试
  │   └─ 否 ↓
  │
  ├─ 用户说"测试XX功能"（单一功能）？
  │   └─ 是 → 单功能测试
  │   └─ 否 ↓
  │
  ├─ 是否发版前全面验证？
  │   └─ 是 → 完整测试
  │   └─ 否 ↓
  │
  ├─ 是否验证代码变更影响？
  │   └─ 是 → 回归测试
  │   └─ 否 ↓
  │
  └─ 是否快速验证核心功能？
      └─ 是 → Smoke 测试
```

### 1.3 选择示例

| 场景 | 推荐类型 | 理由 |
|------|---------|------|
| 新版本部署后验证 | Smoke | 快速确认系统可用 |
| 测试登录功能 | 单功能 | 完整测试登录功能的所有场景 |
| 测试搜索功能 | 单功能 | 完整测试搜索功能的所有场景 |
| 修改了搜索功能 | 回归 | 验证搜索相关功能及依赖 |
| 发版前全面验证 | 完整 | 确保所有功能正常 |
| 订单支付流程测试 | E2E | 跨系统验证（前端→后端→支付网关） |

### 1.4 选择注意事项

- **可叠加**：可以先 Smoke → 通过后再 完整
- **根据时间**：时间紧迫选 Smoke，时间充裕选 完整
- **根据风险**：高风险区域选择更全面的测试

## 2. 各测试类型的测试设计要点

### 2.1 Smoke（冒烟测试）设计要点

**设计原则**：快速验证核心功能可用

**测试重点**：
- ✅ 核心业务流程（登录 → 主要操作 → 登出）
- ✅ 关键页面加载
- ✅ 关键数据展示（总数、统计）
- ❌ 不关注：边界场景、异常处理、数据准确性

**用例设计模式**：
| 用例类型 | 说明 | 示例 |
|---------|------|------|
| 核心流程 | 主流程能否走通 | 登录→搜索→查看详情 |
| 页面加载 | 关键页面能否访问 | 首页、列表页、详情页 |
| 数据展示 | 关键数据是否显示 | 员工总数、部门数量 |

**用例数量**：5-10 个

---

### 2.2 单功能测试设计要点

**设计原则**：完整测试指定功能的所有场景

**测试重点**：
- ✅ 该功能的所有功能点
- ✅ 正向场景
- ✅ 异常场景
- ✅ 边界场景
- ✅ 数据验证（L2 级准确性）

**用例设计模式**：
| 用例类型 | 说明 | 示例 |
|---------|------|------|
| 功能覆盖 | 该功能每个交互点 | 输入框、按钮、链接 |
| 正向场景 | 正常操作流程 | 登录成功、搜索有结果 |
| 异常场景 | 错误处理 | 密码错误、无结果提示 |
| 边界场景 | 极限条件 | 最大长度、空值、超长 |

**用例规模**：按功能复杂度决定（简单功能5-8个，复杂功能15+个）

---

### 2.3 回归测试设计要点

**设计原则**：验证代码变更未影响现有功能

**测试重点**：
- ✅ 本次变更相关的功能
- ✅ 高频使用功能
- ✅ 历史问题区域
- ✅ 可能受影响的功能（依赖关系）
- ⚠️ 选择性：边界场景

**用例设计模式**：
| 用例类型 | 说明 | 示例 |
|---------|------|------|
| 变更功能 | 本次修改的功能 | 搜索功能优化后测试搜索 |
| 高频功能 | 用户经常使用的功能 | 登录、列表加载、筛选 |
| 历史问题 | 之前出过问题的区域 | 之前崩溃的详情页 |
| 依赖功能 | 依赖变更功能的其他功能 | 搜索依赖的数据加载 |

**用例数量**：15-30 个

---

### 2.4 完整测试设计要点

**设计原则**：全面验证所有功能

**测试重点**：
- ✅ 所有功能点
- ✅ 正向场景
- ✅ 异常场景
- ✅ 边界场景
- ✅ 数据验证（L2 级准确性）

**用例设计模式**：
| 用例类型 | 说明 | 示例 |
|---------|------|------|
| 功能覆盖 | 每个功能至少 1 个用例 | 部门筛选、姓名搜索、重置 |
| 正向场景 | 正常操作流程 | 搜索 → 显示结果 |
| 异常场景 | 错误处理 | 无结果、网络错误 |
| 边界场景 | 极限条件 | 最大长度、空值、大数据量 |
| 数据验证 | 数据准确性 | 搜索结果包含关键词、筛选结果正确 |

**用例数量**：50+ 个

**参考章节**：
- §3 按功能类型的测试要点
- §4 边界测试具体方法
- §5 数据验证具体方法

---

### 2.5 E2E（端到端测试）设计要点

**设计原则**：验证跨系统、跨模块的完整业务流程

**测试重点**：
- ✅ 跨系统业务流程（前端 → 后端 → 第三方服务）
- ✅ 用户真实使用场景
- ✅ 数据流转（从创建到展示）
- ⚠️ 不关注：单点功能细节

**用例设计模式**：
| 用例类型 | 说明 | 示例 |
|---------|------|------|
| 跨系统流程 | 跨多个系统的完整流程 | 下单 → 支付 → 库存扣减 → 物流 |
| 用户场景 | 模拟真实用户操作 | 新用户注册 → 完善资料 → 首次购买 |
| 数据流转 | 验证数据在各系统正确传递 | 订单创建 → 支付 → 订单状态更新 |

**用例数量**：10-20 个

**示例**：
```markdown
### E2E-001: 完整购物流程

**测试场景**：用户从浏览商品到支付成功的完整流程

**涉及系统**：
- 前端：商品浏览、购物车
- 后端：订单服务
- 第三方：支付网关

**测试步骤**：
1. 登录系统
2. 浏览商品列表
3. 添加商品到购物车
4. 进入结算页面
5. 选择支付方式
6. 完成支付
7. 验证订单状态更新
8. 验证库存扣减
```

---

### 2.6 测试类型设计要点对比

| 维度 | Smoke | 单功能 | 回归 | 完整 | E2E |
|------|-------|--------|------|------|-----|
| **核心流程** | ✅ 必须 | ✅ 必须 | ⚠️ 选择性 | ✅ 必须 | ✅ 必须 |
| **正向场景** | ✅ 必须 | ✅ 必须 | ✅ 必须 | ✅ 必须 | ✅ 必须 |
| **异常场景** | ❌ 不关注 | ✅ 必须 | ⚠️ 选择性 | ✅ 必须 | ⚠️ 选择性 |
| **边界场景** | ❌ 不关注 | ✅ 必须 | ⚠️ 选择性 | ✅ 必须 | ❌ 不关注 |
| **数据验证** | ⚠️ 存在性 | ✅ 准确性 | ⚠️ 选择性 | ✅ 准确性 | ⚠️ 一致性 |
| **历史问题** | ❌ 不关注 | ⚠️ 选择性 | ✅ 必须 | ✅ 必须 | ❌ 不关注 |
| **跨系统** | ❌ 不关注 | ❌ 不关注 | ❌ 不关注 | ⚠️ 选择性 | ✅ 必须 |

**图例**：✅必须 / ⚠️选择性 / ❌不关注

> **重要说明**：完整测试应包含跨系统场景。当应用涉及多个系统或模块集成时（如前端→后端→第三方服务），完整测试应设计跨系统验证用例，确保数据流转和业务流程的完整性。

## 3. 功能点识别方法论

### 3.1 基于用户需求的功能提取

**原则**：用户需求优先，代码库分析为辅

**操作步骤**：
1. 询问用户测试目标和范围
2. 让用户描述需要测试的功能模块
3. 记录用户提到的关键功能点
4. 仅在用户描述不明确时，才查看代码库补充

**输出格式**：
```markdown
## 识别的功能点

### 用户需求描述
[用户原始描述]

### 提取的功能点
1. 功能点A - [说明]
2. 功能点B - [说明]
3. 功能点C - [说明]
```

### 3.2 基于页面侦测的元素识别

**侦测策略**：
1. **基础快照**：获取初始页面快照
2. **多轮快照**（针对动态内容）：
   - 滚动页面后再次快照
   - 触发下拉/展开后再次快照
   - 等待异步加载后再次快照
3. **特殊元素侦测**：
   - iframe：检查并进入 iframe 内部侦测
   - 弹窗：触发弹窗后侦测
   - 动态加载：等待加载完成后侦测

**元素分类**：
| 元素类型 | 侦测方法 | 测试考虑 |
|---------|---------|---------|
| 按钮 | getByRole('button') | 点击行为、禁用状态 |
| 输入框 | getByPlaceholder/Label | 输入验证、边界值 |
| 下拉框 | getByRole('combobox') | 选项验证、默认值 |
| 链接 | getByRole('link') | 跳转验证、新窗口 |
| 表格 | 定位器 + 行/列 | 数据验证、排序、分页 |

### 3.3 功能点分类模板

**按业务功能分类**：
```markdown
### 核心业务功能
- 登录/认证
- 主要业务流程（如搜索、下单）

### 辅助功能
- 设置/配置
- 帮助/文档

### 数据展示功能
- 列表/表格
- 统计/图表

### 数据操作功能
- 表单提交
- 导入/导出
```

## 5. 边界测试具体方法

### 5.1 数值边界

**测试方法**：
| 边界类型 | 测试值 | 预期结果 |
|---------|--------|---------|
| 最小值 | min-1, min, min+1 | 验证边界处理 |
| 最大值 | max-1, max, max+1 | 验证边界处理 |
| 零值 | 0 | 验证零值处理 |
| 负值 | -1, -100 | 验证负值处理 |
| 极值 | 接近类型极限 | 验证极值处理 |

**示例**：
- 年龄输入框（范围 1-120）：测试 0, 1, 2, 119, 120, 121
- 金额输入（范围 0.01-99999.99）：测试 0, 0.01, 0.02, 99999.98, 99999.99, 100000

---

### 5.2 字符串边界

**测试方法**：
| 边界类型 | 测试值 | 预期结果 |
|---------|--------|---------|
| 空字符串 | "" | 验证空值处理 |
| 单字符 | "a" | 验证最小输入 |
| 最大长度 | 刚好max长度 | 验证正常接受 |
| 超长字符串 | max+1长度 | 验证拒绝或截断 |
| 特殊字符 | !@#$%^&*() | 验证特殊字符处理 |
| SQL注入 | "'; DROP TABLE--" | 验证安全性 |
| XSS | "<script>alert(1)</script>" | 验证安全性 |

**示例**：
- 用户名（最长20字符）：测试 "", "a", "a"*20, "a"*21
- 备注（最多500字符）：测试 "", "a", "a"*500, "a"*501

---

### 5.3 数据量边界

**测试方法**：
| 边界类型 | 测试数据 | 预期结果 |
|---------|---------|---------|
| 空数据 | 0 条 | 显示空状态 |
| 单条数据 | 1 条 | 正常显示 |
| 分页边界 | 刚好整页、多1条 | 验证分页逻辑 |
| 大数据量 | 1000+ 条 | 验证性能和显示 |

**示例**：
- 列表分页（每页10条）：测试 0 条, 1 条, 10 条, 11 条, 1000 条
- 搜索结果：测试无结果、单结果、大量结果

## 6. 数据验证具体方法

### 6.1 验证级别定义

| 级别 | 定义 | 适用场景 | 示例 |
|------|------|---------|------|
| **L1** | 存在性验证 | Smoke 测试 | 元素存在、页面加载 |
| **L2** | 准确性验证 | 功能/完整测试 | 数值正确、文本准确 |
| **L3** | 业务规则验证 | 完整测试 | 符合业务逻辑计算 |

---

### 6.2 统计卡片验证

**功能特征**：显示数字、百分比、趋势图标

**验证要点**：
| 验证项 | 验证方法 | 示例 |
|-------|---------|------|
| 数值存在 | 元素可见性 | 统计数字显示 |
| 数值准确性 | 对比预期值 | "共 941 人" = 实际行数 |
| 百分比准确性 | 计算验证 | 增长率 = (新值-旧值)/旧值 × 100% |

**实施步骤**：
1. 获取页面显示的数值
2. 获取实际数据（如列表行数）
3. 对比验证

---

### 6.3 数据表格验证

**功能特征**：列表、分页、排序

**验证要点**：
| 验证项 | 验证方法 | 示例 |
|-------|---------|------|
| 行数准确性 | 对比显示与实际 | "共 941 人" = 实际行数 |
| 字段值准确性 | 抽样验证 | 随机验证 3-5 行数据 |
| 分页准确性 | 验证分页逻辑 | 每页条数、总页数 |

**实施步骤**：
1. 获取表格显示的总行数
2. 实际计算表格行数
3. 对比验证
4. 抽样验证字段值

---

### 6.4 搜索/筛选验证

**功能特征**：搜索框、搜索按钮、筛选条件、结果列表

**验证要点**：
| 验证项 | 验证方法 | 示例 |
|-------|---------|------|
| 搜索结果包含关键词 | 每行检查 | 每行姓名包含"王" |
| 筛选结果符合条件 | 逐项验证 | 部门 = "财务部" |

**实施步骤**：
1. 执行搜索/筛选
2. 获取所有结果行
3. 逐行验证包含关键词或符合条件

---

### 6.5 图表验证

**功能特征**：ECharts、Chart.js、SVG 图表

**验证要点**：
| 验证项 | 验证方法 | 示例 |
|-------|---------|------|
| 数据点准确性 | 对比数据源 | 图表数值 = 原始数据 |
| 系列值准确性 | 系列求和验证 | 各系列和 = 总数 |
| 轴标签准确性 | 标签正确性 | X轴/Y轴标签正确 |

**实施步骤**：
1. 获取图表数据（通过 API 或页面解析）
2. 对比预期数据
3. 验证数据点、系列值、轴标签

**实施步骤**：

## 9. 需求覆盖分析方法

### 9.1 需求-测试点映射

**核心原则**：优先基于用户需求，其次基于代码库分析

**操作流程**：

1. **需求提取**
   - 从用户描述中提取功能需求
   - 记录需求的优先级（用户明确标注）

2. **需求分解**
   - 将每个需求分解为可测试的测试点
   - 使用"用户故事"格式：作为[角色]，我想要[功能]，以便[目的]

3. **需求-测试点映射表**
   | 需求ID | 需求描述 | 测试点 | 测试用例 | 状态 |
   |--------|---------|--------|---------|------|
   | REQ-001 | 用户可以搜索员工 | 姓名搜索功能 | TC-003 | ✅ |
   | REQ-001 | 用户可以搜索员工 | 部门筛选功能 | TC-002 | ✅ |
   | REQ-002 | 用户可以查看员工详情 | 详情弹窗显示 | TC-005 | ✅ |

### 9.2 需求覆盖率计算

```
需求覆盖率 = 已覆盖需求数 / 需求总数 × 100%
```

**目标标准**：
| 测试类型 | 需求覆盖率目标 |
|---------|---------------|
| Smoke | ≥ 80% (核心需求) |
| 单功能 | 100% (该功能相关需求) |
| 回归 | ≥ 90% (变更相关需求) |
| 完整 | 100% (所有需求) |

### 9.3 需求覆盖完整性判断

**检查项**：
- [ ] 每个需求至少有1个测试点
- [ ] 核心需求（用户标注的）有详细测试
- [ ] 需求的所有约束条件都有测试
- [ ] 需求的所有边界条件都有测试

### 9.4 侦测结果的使用

**侦测的作用**：
- ✅ **辅助验证**：验证需求中提到的功能确实存在
- ✅ **补充遗漏**：发现用户未明确提到但存在的功能
- ✅ **元素定位**：为测试点提供可操作的元素选择器

**侦测不是**：
- ❌ 需求的主要来源
- ❌ 测试范围的唯一依据

**使用流程**：
```
1. 用户需求 → 提取需求列表
2. 侦测页面 → 验证需求功能存在
3. 对比分析 → 发现遗漏功能
4. 确认用户 → 是否需要补充遗漏功能的测试
```

## 10. 测试数据规划

### 10.1 测试数据类型分类

| 数据类型 | 说明 | 示例 | 优先级 |
|---------|------|------|--------|
| **认证数据** | 登录凭证、API密钥 | 用户名/密码、Token | P0 |
| **基础数据** | 正常功能运行的数据 | 有效用户名、正常搜索词 | P0 |
| **边界数据** | 边界值测试的数据 | 最大长度、最小值、0值 | P1 |
| **异常数据** | 异常场景测试的数据 | 空值、特殊字符、超长 | P1 |
| **业务数据** | 业务逻辑验证的数据 | 特定业务规则的数据 | P1 |

### 10.2 测试数据来源决策

**决策原则**：

| 数据类型 | 优先来源 | 备选来源 |
|---------|---------|---------|
| 认证数据 | **用户提供** | 测试账号 |
| 基础数据 | AI生成 | 用户提供 |
| 边界数据 | AI生成 | - |
| 异常数据 | AI生成 | - |
| 业务数据 | **用户提供** | AI估算 |

**原因**：
- 认证数据：必须用户提供（真实凭证）
- 业务数据：用户最了解业务规则
- 其他数据：AI可以根据需求生成

### 10.3 与用户交互的时机和方式

**时机1：步骤1 需求分析阶段**

询问内容：
1. **认证数据**（如有）
   - 是否有测试账号？
   - 是否需要保存认证状态？

2. **业务数据**（如适用）
   - 是否有特定的业务数据需要测试？

3. **数据来源确认**（使用 AskUserQuestion）
   - "我将提供测试数据（CSV/JSON文件或详细描述）"
   - "由AI生成测试数据"
   - "混合：认证数据我提供，其他AI生成"
