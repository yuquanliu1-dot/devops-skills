# Web 自动化测试技能改进设计

**日期**: 2026-03-29
**版本**: 1.1.0
**状态**: 设计中

## 概述

改进 `web-auto-testing` 技能的用户交互体验和执行可靠性。解决以下问题：

1. **报告生成时机不当**：执行部分用例后就生成报告
2. **AskUserQuestion 使用混乱**：开放性输入和结构化选择不分
3. **需求来源询问缺失**：未询问是否有需求文档
4. **需求澄清流程不明确**：需求不明确时缺少操作指引

## 设计目标

- 执行完所有用例后再生成报告
- 明确区分直接对话和 AskUserQuestion 的使用场景
- 添加需求文档来源确认
- 建立需求澄清决策流程

---

## 一、智能询问逻辑架构

### 需求明确度评估

在步骤 1 开始时执行评估，判断是否需要询问。

```
输入源：用户初始请求、需求文档（如有）、代码库（可选）

评估维度：
- 测试 URL         → 已明确?
- 测试范围         → 已明确?
- 测试类型         → 已明确?
- 认证信息         → 已明确?
- 测试数据         → 已明确?
```

### 优先级规则

1. **需求文档 > 用户口头描述 > 代码库分析**
2. **明确信息 > 默认假设**

### 关键词识别

| 用户关键词 | 自动推断为 |
|-----------|-----------|
| "全面测试"、"完整测试" | 完整测试 |
| "快速验证"、"冒烟" | Smoke 测试 |
| "回归"、"改了XX" | 回归测试 |
| "端到端"、"全流程" | E2E 测试 |

### 示例场景

| 场景 | 行为 |
|------|------|
| 用户说"全面测试员工通信录" | 跳过测试类型询问，默认"完整测试" |
| 用户提供需求文档说"Smoke 测试" | 按文档执行，不询问 |
| 用户只说"测试这个应用" | 询问 URL、测试范围、测试类型 |
| 用户说"测试登录功能" | 询问 URL、认证信息；跳过测试范围询问 |

## 二、交互工具决策表

### 直接对话（开放性输入）

| 问题 | 示例 |
|------|------|
| 测试 URL | "请提供测试应用的 URL" |
| 功能描述 | "请描述需要测试的功能模块" |
| 认证凭证 | "请提供测试账号密码（如有）" |
| 业务数据 | "请提供测试数据（如有）" |
| 自由反馈 | "还有其他需求吗？" |

### AskUserQuestion（结构化选择）

| 问题 | 选项 |
|------|------|
| 测试类型选择 | Smoke/回归/完整/E2E（4 选 1） |
| 数据来源确认 | 用户提供/AI 生成/混合（3 选 1） |
| 认证状态保存 | 是/否（2 选 1） |
| 测试计划确认 | 确认/调整/补充/跳过（4 选 1） |
| 执行完整性检查 | 继续/跳过（2 选 1） |
| 文档生成确认 | 生成计划+用例/仅计划/调整/执行（4 选 1） |

### 智能跳过规则

- 用户已明确 → 跳过询问
- 需求文档指定 → 按文档执行，跳过询问
- 关键词识别 → 自动推断

## 三、需求澄清决策树

```
开始：收集初始需求
    ↓
需求文档？ ──是→ 按文档执行，跳过澄清
    ↓否
用户描述？ ──空值→ 询问：你想测试什么？
    ↓有内容
关键词识别 ──→ "全面测试"→完整，"快速"→Smoke
    ↓
信息完整？
    ├── 不完整 → 只询问缺失的信息
    └── 完整 → 展示确认（AskUserQuestion）
```

### 澄清策略表

| 缺失信息 | 澄清方式 | 工具 |
|---------|---------|------|
| 测试 URL | 直接询问 | 直接对话 |
| 测试范围 | 询问：哪些模块/功能？ | 直接对话 |
| 测试类型 | 关键词识别 + 询问 | AskUserQuestion |
| 认证信息 | 询问：需要登录吗？ | AskUserQuestion |
| 测试数据 | 询问：数据来源？ | AskUserQuestion + 验证流程 |

## 四、测试数据验证流程

### 输入来源和格式

| 来源 | 支持情况 |
|------|---------|
| 本地文件路径 | ✅ `C:\data\test.csv` |
| 网络地址 | ✅ `https://example.com/data.json` |
| 文档链接 | ✅ Notion/飞书/语雀（通过 web-reader） |
| 直接粘贴 | ✅ 用户在对话中粘贴 |

| 格式 | 支持情况 |
|------|---------|
| CSV | ✅ 表格数据 |
| JSON | ✅ 结构化数据 |
| Excel | ✅ .xlsx 文件 |
| Markdown | ✅ 文档格式，含表格和描述 |

### 数据验证流程

**触发时机**：用户在 AskUserQuestion 中选择"用户提供测试数据"并提交数据地址/内容后，立即执行以下流程：

```
1. 收集阶段
   - 询问数据文档地址/直接粘贴数据

2. 分析阶段
   - 解析数据结构
   - 对比测试需求，检查缺失
   - 生成数据完整性报告

3. 缺失处理（AskUserQuestion）
   - A：用户提供补充数据
   - B：AI 生成缺失数据
   - C：按现有数据执行（可能覆盖不全）
```

### 数据完整性检查维度

| 数据类型 | 检查项 | 示例 |
|---------|-------|------|
| 认证数据 | 账号/密码完整性 | 缺少密码 |
| 业务数据 | 数量是否足够 | 需要 10 个商品，只有 5 个 |
| 边界数据 | 是否包含极值 | 缺少最大/最小值 |
| 异常数据 | 是否包含异常场景 | 缺少空值、特殊字符 |

## 五、执行流程完整性保证

### 执行跟踪对象

```javascript
执行跟踪 = {
  总用例数: 15,
  已执行: 0,
  通过: 0,
  失败: 0,
  跳过: 0,
  失败用例列表: [
    { 用例名: "登录-错误密码", 截图: "login-failed.png" }
  ],
  开始时间: timestamp,
  状态: "执行中" | "已完成" | "部分执行"
}
```

### 执行流程

```
1. 初始化执行跟踪

2. 逐个执行循环
   FOR 每个测试用例：
     ├─ 显示进度："[${已执行+1}/${总用例数}] ${用例名称}"
     ├─ 执行测试用例
     ├─ 更新执行跟踪
     └─ 失败时截图保存
   END FOR

3. 完整性检查（强制）
   IF 已执行 != 总用例数:
     ── AskUserQuestion ──
     "⚠️ 未执行完所有用例（${已执行}/${总用例数}）"
     选项：继续执行 / 跳过生成报告
   END IF

   **用户选择"跳过生成报告"后的处理**：
   - 设置执行跟踪.状态 = "部分执行"
   - 保存当前执行跟踪到 artifacts/execution-progress.json
   - 询问用户：是否保存进度以便稍后恢复？
     - 是 → 保存进度，可稍后恢复执行
     - 否 → 结束本次测试
```

### 进度显示规范

- 每个用例执行前：显示进度
- 每 5 个用例：显示汇总
- 执行完成：显示最终统计

## 六、报告生成与错误处理

### 报告生成前检查（强制）

```
IF 已执行 < 总用例数:
  ── 禁止生成"完整"报告 ──
  ── AskUserQuestion ──
  "测试未执行完整（${已执行}/${总用例数}）"
  选项：执行剩余用例 / 生成不完整报告
ELSE:
  ── 生成完整报告 ──
END IF
```

### 失败用例处理（保留原有设计）

```javascript
IF 测试失败:
  调用 browser_take_screenshot
  保存至: artifacts/screenshots/{feature}-{case}-failed.png
  记录文件名到执行跟踪
END IF
```

### 报告内容

| 内容项 | 说明 |
|-------|------|
| 执行摘要 | 总数/已执行/通过/失败/跳过 |
| 执行状态标记 | 完整/部分执行 |
| 覆盖率指标 | 功能/元素/场景覆盖率 |
| 失败测试详情 | 错误、截图链接、重现步骤、修复建议 |
| 部署决策 | 基于成功指标判断 |

### 错误恢复策略

| 错误类型 | 恢复策略 | 用户交互 |
|---------|---------|---------|
| 数据缺失 | 询问补充/AI 生成/跳过 | AskUserQuestion |
| 应用无法访问 | 3 次重试 | 询问继续/终止 |
| 选择器失效 | 记录并跳过，继续执行 | 无需询问 |
| 执行中断 | 保存进度 | 询问恢复/终止 |

## 附录：核心数据流

```
步骤 1 需求分析
  ├─→ 需求明确度评估 ──→ 全部明确? ──是→ 跳过询问
  │                              │
  │                             否
  │                              │
  └─→ 只询问缺失信息 ──→ 关键词识别 ──→ 推断测试类型
          │
          ├─→ 测试数据来源? ──→ 用户提供? ──是→ 数据验证
          │                                  │
          │                                 缺失?
          │                                  │
          │                                  是→ Ask 处理
          │
          └─→ 确认计划 ──→ AskUserQuestion

步骤 4 测试执行
  ├─→ 初始化执行跟踪
  ├─→ FOR 每个用例: 显示进度 → 执行 → 失败?截图 → 更新跟踪
  └─→ 完整性检查 ──→ 未完整? ──是→ AskUserQuestion

步骤 5 报告生成
  └─→ 最终检查 ──→ 完整? ──是→ 生成报告
                          │
                         否
                          │
                          → AskUserQuestion
```
